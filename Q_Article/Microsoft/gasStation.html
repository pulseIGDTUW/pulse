Gas Station — Interview Walkthrough
Problem: Given arrays gas[] and cost[], find the start index to complete the circular tour (or -1). — Interview flow: brute-force → optimize → one-pass
Interviewer: Here’s the problem — you have two arrays gas and cost. At station i you get gas[i] and it costs cost[i] to reach the next station. Starting with empty tank, which station index allows a full circular tour? If none, return -1.
Candidate: Understood. A quick check: the problem guarantees uniqueness of the solution if it exists, right?
Interviewer: Yes — if there is a solution, it is unique. Now walk me through how you'd approach this in an interview.

Step 1 — Brute force (talk it out first)

Candidate: The simplest idea is to try every station as a starting point and simulate the trip. For each start s, keep a running tank and move around the circle. If tank ever drops below zero, this start fails — try next start.
Interviewer: That’s a correct starting idea. Implement it if we had to — but let's discuss complexity first.

Complexity (brute force): O(n²) time — because for each of n starts you might traverse up to n stations. O(1) extra space.

// Brute-force: try every index as starting point int startStation(vector<int> &gas, vector<int> &cost) { int n = gas.size(); int startIdx = -1; for (int i = 0; i < n; ++i) { int tank = 0; bool ok = true; for (int j = 0; j < n; ++j) { int idx = (i + j) % n; tank += gas[idx] - cost[idx]; if (tank < 0) { ok = false; break; } } if (ok) { startIdx = i; break; } } return startIdx; }

Step 2 — Observe and optimize (interviewer nudges)

Interviewer: The brute force will pass small tests but not large ones. Can you improve it?
Candidate: Think greedily — walk through stations once keeping a running available gas. If available becomes negative at station i, then any start between our current candidate and i cannot work. So we move the start to i+1 and reset running gas.
Interviewer: Why can we safely skip all starts from the previous candidate to i?
Candidate: Because starting at any station s between old start and i will have even less gas upon reaching i than the old start — so those cannot succeed if the old start failed at i.

Greedy (two-pass verification)

// Greedy: find candidate start, then verify by circular check int startStation(vector<int> &gas, vector<int> &cost) { int n = gas.size(); int start = 0; int curr = 0; for (int i = 0; i < n; ++i) { curr += gas[i] - cost[i]; if (curr < 0) { start = i + 1; // reset candidate curr = 0; } } // verify that start works by simulating once curr = 0; for (int i = 0; i < n; ++i) { int idx = (start + i) % n; curr += gas[idx] - cost[idx]; if (curr < 0) return -1; } return start; }
Interview tip: In the first pass we find a candidate. The second pass guarantees that total gas is enough — this two-step thinking is what interviewers expect you to explain.

Step 3 — Final refinement: one-pass greedy (optimal)

Candidate: We can combine the verification into the same pass by tracking the total net gas. If total net (sum of gas[i]-cost[i]) is < 0, return -1 immediately — otherwise the candidate found in the single pass is valid.
// One-pass greedy: track total and current gas int startStation(vector<int> &gas, vector<int> &cost) { int n = gas.size(); int start = 0; int curr = 0; int total = 0; for (int i = 0; i < n; ++i) { int diff = gas[i] - cost[i]; curr += diff; total += diff; if (curr < 0) { curr = 0; start = i + 1; } } if (total < 0) return -1; return start; }

Why this is correct

  • If total < 0, sum of gas is insufficient to cover sum of costs — impossible to finish anywhere.
  • The candidate start is the earliest index after the last failing point; by the previous greedy argument no index before that could work.
  • Uniqueness follows because net total determines feasibility and greedy picks the correct start.

ApproachTimeSpace
Brute-forceO(n²)O(1)

Greedy (one-pass)O(n)O(1)

Key takeaways — what to say in an interview

  • Start with brute force, then optimize: explain the naive idea, its cost, and why we need better. Interviewers love this progression.
  • Explain the greedy observation: when current available gas becomes negative at i, any start between the old start and i fails — justify this with an inequality argument.
  • Use total net gas: explicitly state the total check (sum(gas)-sum(cost)) — if negative, no solution exists.
  • State edge cases: single station, all zeros, negative diff at last index — show how your algorithm handles them.
  • Talk complexity clearly: time and space for each step, and why one-pass is optimal.
Small rehearsal script: "I would try brute-force first (correct but slow), then show the greedy argument that lets us skip many starts, and finally combine verification into one pass using a total-sum check." — say this succinctly in an interview.
